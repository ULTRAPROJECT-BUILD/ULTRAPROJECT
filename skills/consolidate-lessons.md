# Consolidate Lessons

You are synthesizing accumulated lessons and decisions into the system's working knowledge. This prevents the vault from becoming a graveyard of unread insights.

## When to Run

- After every 10 new lessons accumulate (platform-wide)
- After a client project completes and is archived
- When the orchestrator detects repeated failures in the same area
- On a periodic operator-initiated cadence during idle time

## Process

### Step 1: Gather Lessons and Decisions

1. Glob all lesson files:
   - Platform: `vault/lessons/*.md`
   - Client-scoped: `vault/clients/*/lessons/*.md`
2. Glob all decision files:
   - Platform: `vault/decisions/*.md`
   - Client-scoped: `vault/clients/*/decisions/*.md`
3. Read the frontmatter and body of each.
4. Group by tags, project, and domain.

### Step 2: Identify Patterns

For each group, look for:

- **Repeated lessons** — the same insight learned multiple times means it's not embedded in the system yet.
- **Contradictory decisions** — two decisions that conflict indicate an unresolved policy question.
- **Cluster themes** — 3+ lessons about the same topic (e.g., "email deliverability", "client communication", "API rate limits") indicate a knowledge area that should be codified.

### Step 3: Record Insights for Skills (DO NOT MODIFY PLATFORM SKILLS)

For each pattern found:

1. Identify which skill would benefit from the lesson.
2. **If it's a platform skill (`skills/*.md`):** Do NOT modify it. Instead, save the insight as a lesson note in `vault/lessons/` with the tag `skill-improvement` and note which skill it applies to. The admin will review and apply during collaboration sessions.
3. **If it's an archived skill (`vault/archive/skills/`):** You may update it directly with the insight. Add a comment noting the source: `(Learned from {lesson-slug}, {date})`
4. Do NOT remove the original lesson file — it's part of the audit trail.

### Step 4: Create or Update Patterns

If 3+ lessons cluster around the same theme and no pattern file exists:

1. Create a new pattern at `vault/archive/patterns/{slug}.md` using the pattern schema from [[SCHEMA]].
2. Set confidence based on how consistent the lessons are.
3. Reference the source lessons.

If a pattern already exists for this theme:
1. Update `observed_count` and `confidence`.
2. Add new source references.

### Step 4b: Pattern Candidate Processing

Check for lessons with `subtype: pattern-candidate` (generated by [[meta-improvement]]).

1. Gather all pattern-candidate lessons not yet processed (no `status: processed` in frontmatter).
2. **Exclusion:** These lessons must NOT be included in the general Step 2-4 clustering. Steps 1-4 should skip lessons with `subtype: pattern-candidate` — they are processed exclusively here to avoid double-counting.
3. Process them with the same theme-based clustering as regular lessons (Step 4 rules). The `failure_category` field is metadata for filtering and reporting, NOT the clustering key — three unrelated candidates in the same category do not automatically merge. Cluster by shared root cause and proposed remediation.
3. If 3+ pattern-candidate lessons cluster on the same theme/root cause, create a new pattern following existing conventions (Step 4 rules apply).
4. Mark processed pattern-candidate lessons with `status: processed` in their frontmatter.
5. Include in the consolidation report: pattern candidates processed, patterns created from candidates, candidates still pending (fewer than 3 in cluster).

### Step 5: Flag Contradictions

If contradictory decisions are found:
1. Create a ticket assigned to `human` describing the contradiction.
2. Reference both decisions.
3. Do NOT resolve contradictions autonomously — these need human judgment.

### Step 6: Report

Return:
- **Lessons processed:** count
- **Decisions processed:** count
- **Skill files updated:** list with what changed
- **Patterns created/updated:** list
- **Contradictions flagged:** list
- **Redundant lessons:** lessons that repeat an already-codified insight (candidates for cleanup)

## Principles

- **Consolidation is additive** — improve skill files, don't rewrite them. Add error handling cases, refine process steps, note edge cases.
- **Preserve the audit trail** — original lessons and decisions stay in the vault. Consolidation references them, it doesn't replace them.
- **Humans resolve contradictions** — if two decisions conflict, escalate. Don't pick a winner.
- **Dry run first** — when in doubt, run with `dry_run: true` to see what would change before committing.
